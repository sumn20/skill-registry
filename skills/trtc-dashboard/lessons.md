## 1. API 事件缺失导致误判"疑似没讲话"
- **场景**: 发送端有 mute/unmute 操作，但 API detail_event 中缺少"停止采集音频"事件
- **错误结论**: "疑似没讲话"
- **正确结论**: 需查看页面详细事件才能确认，本案为"接收端系统播放设备音量为0"
- **经验规则**: API 事件不完整时，不要仅凭 API 数据下结论。页面"查看详细事件"中的事件列表比 API 更全，尤其是"停止采集音频"、"麦克风软件音量=0"、"系统播放设备音量=0"等关键事件
- **日期**: 2026-03-13

## 2. mute 判定阈值过严导致漏判
- **场景**: 发送端 has_mute=True，采集音量 avg=766.7（500~2000 区间）
- **错误结论**: "疑似没讲话"（mute 判定要求 cap_avg<500，不满足）
- **正确结论**: "mute 静音"（500~2000 区间在 mute 场景下应判为 mute 静音）
- **经验规则**: mute flag 为 True 时，采集音量在 500~2000 区间属于 mute 静音，不应跳过进入"疑似没讲话"
- **日期**: 2026-03-13
